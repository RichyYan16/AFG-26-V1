# Algorithm Specification vs Implementation Review

## Your Specification (RAW BASIC ALGORITHM)

```
1. User clicks "I'M STUCK"
2. Questionnaire intro screen (animation + explanation)
3. 5 base questions administered
4. Word Embedding Algorithm:
   - Takes responses
   - Returns: vector representation [a, b, c, ...]
5. Classifier:
   - Turns vector into Stuck Type prediction
6. Gemini API (5 internal iterations):
   - Analyzes stuck type
   - Generates 1 follow-up question per iteration
   - User responds to each
7. Generate Diagnosis:
   - Uses: Gemini API + embedding vector matrix
   - Format: Confidence levels for each stuck type (high→low)
   - Brief summary
   - Button: "Do you want to change this?"
8. Intervention Plans Page:
   - Loads (takes a second)
   - Generated by Gemini API
   - List of plan ideas
```

## Current Implementation Review

### ❌ What Needs Adjustment

1. **Vector Output**: Currently returns normalized scores (0-1). Need to return raw embedding vector [a, b, c, ...] from embeddings

2. **5 Internal Iterations**: Currently optional/implicit. Need to make it **explicit and required** - always generate 5 follow-up questions internally

3. **Vector Matrix**: Currently combines embedding scores + Gemini. Need to ensure we're using the "overall combined matrix from word embedding" as the basis

4. **Question Flow**: Currently shows 5 base Qs, then optionally follow-ups. Need to clarify if follow-ups are shown to user or internal only

5. **Diagnosis Display Format**: Need exact format with:
   - Confidence levels for each type (high to low)
   - Very brief summary
   - "Do you want to change this?" button

6. **Intervention Plans**: Currently generates one. Need to generate **multiple plan ideas** as a list

7. **Loading State**: Need explicit "loading" state that takes ~1-2 seconds (simulated if needed)
